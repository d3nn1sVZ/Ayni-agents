## Problem Statement

Spiral pide construir un servicio donde agentes de IA paguen o cobren con Lightning Network, demostrando "money moves" y idealmente habilitando algo imposible con tarjetas o stablecoins.

El equipo identificó dos productos candidatos en el meeting del 2026-04-25 14:02 GMT-05. Resolvimos la tensión via /office-hours: ganó la idea de **plugin colaborativo de tribu**, donde comunidades especializadas (ej. economistas, abogados tributarios, científicos de datos) construyen y mantienen un plugin que otros agentes consumen vía API, y las comisiones se distribuyen automáticamente entre todos los contribuidores.

## What Makes This Cool

El "money moment" no es que un agente pague otro. Es que un agente paga **un plugin colectivo y 50 humanos reciben su parte automáticamente en el mismo segundo**. Eso es imposible con Stripe (fees > monto). Eso es lo que el jurado quiere ver.

Es Wikipedia + Patreon + Lightning. Es la primera vez que la inteligencia colectiva puede cobrar como inteligencia colectiva.

## Premises (riesgo asumido en cada una)

1. **L402 + MoneyDevKit puede integrarse en <4h.** Riesgo: medio. Mitigación: si falla, pivotar a plan B (World Bank challenge) — el equipo ya acordó este punto de control.
2. **Lightning permite redistribución programable a N wallets en una sola tx lógica.** Riesgo: técnico. Mitigación: usar Spark (L2 sobre Lightning, mencionado en el reto) o un splitter naive (recibir → disparar N pagos onward).
3. **Los jueces premiarán "redistribución colectiva" como el bonus criterion de "Lightning enables impossible".** Riesgo: bajo. Es la lectura literal del criterion.
4. **El equipo puede demostrar money-moves end-to-end en mainnet con BTC real.** Riesgo: dependiente de Spiral cargando la wallet a tiempo. Mitigación: pedir confirmación a Spiral asap, paralelo desarrollar contra testnet.
5. **Una "tribu seed" puede armarse en horas con datos sintéticos creíbles.** Riesgo: bajo. Para demo basta con 3-5 contribuidores ficticios con wallets reales.

## Approaches Considered

### Approach A: Plugin de tribu colaborativa (CHOSEN)
Comunidades crean/mantienen plugins colectivos. Otros agentes consumen vía API con paywall L402. Cada pago se redistribuye automáticamente entre los contribuidores que mantienen el plugin actualizado. Lightning hace trivial lo que Stripe hace imposible.

### Approach B: Agente personal monetizado (REJECTED)
Cada persona expone su agente personal vía API. Brands pagan per-token. Demo simple pero levanta cejas del jurado por privacidad y NO clava el bonus criterion (Stripe podría hacerlo, solo más caro).

### Approach C: Síntesis (DEFERRED to pitch narrative)
Agentes personales como contribuidores del plugin de tribu. Visión 10/10, feasibility 3/10 en 14h. **Reservada para el video de negocio: "esto es hacia donde escala — cada persona como una tribu de uno."**

## Recommended Approach (Approach A)

### Demo flow (60 segundos del video técnico)

1. **0:00-0:10** Hook: "Un agente de IA necesita saber legislación tributaria peruana. Hoy: scrapea blogs viejos. Mañana: paga sats a una tribu de expertos."
2. **0:10-0:20** Pantalla: agente cliente llama `GET /tribu/tributario-pe/consulta?q=...` → recibe `402 Payment Required` con invoice Lightning.
3. **0:20-0:30** Agente paga 100 sats vía Lightning. Server responde con la respuesta experta.
4. **0:30-0:50** **El money shot**: dashboard muestra los 5 contribuidores de "Tributario PE" recibiendo su parte (40 sats al curador, 30 al validador, 10 cada uno a 3 contribuidores) — todo en el mismo segundo, en tiempo real.
5. **0:50-1:00** Cierre: "Stripe cobra 30¢ mínimo por tx. Esto repartió 100 sats entre 5 wallets. Imposible sin Lightning."

### Arquitectura mínima

```
[Agente cliente]  ──HTTP──>  [Plugin endpoint protegido por L402 paywall]
                                   │
                                   ├── Validate payment via L402
                                   ├── Return plugin response
                                   └── Trigger payout split
                                        │
                                        ├──> Wallet curador  (40%)
                                        ├──> Wallet validator (30%)
                                        └──> 3x Wallet contribuidor (10% c/u)
```

### Stack técnico

- **Frontend:** Next.js 14 + Tailwind (Vercel deploy). Pantalla principal: dashboard de tribus + vista de pagos en tiempo real.
- **Backend:** API routes de Next.js. Integración con MoneyDevKit (`npx @moneydevkit/create`) que ya soporta L402 nativo y wallets de agente.
- **Pago + redistribución:** evaluar Spark (`npx @buildonspark/create-spark-app`) primero — diseñado para programmable payments en L2 sobre Lightning. Si Spark complica, fallback a splitter naive con MDK + N pagos secuenciales.
- **Datos del plugin:** JSON estático con 1-2 tribus seed (Tributario PE, Data Science ES). Respuestas pueden venir de un LLM con system prompt especializado — el "valor" del plugin es el prompt + curación, no la magia.
- **Visualización money-moves:** WebSocket o Server-Sent Events para que el dashboard se actualice en tiempo real cuando llegan pagos.

### Trust & Safety layer (rol de Cindy)

- Anonimización: contribuidores aparecen como `@curador-3a8f`, no nombres reales.
- Reputation seed: cada tribu tiene rating fictício pero coherente (4.7★, 234 consultas).
- Data layer: storytelling sobre por qué la tribu es confiable (ej. "validada por 12 abogados activos").
